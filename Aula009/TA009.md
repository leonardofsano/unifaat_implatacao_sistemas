## 1. Conceitos Fundamentais

### Amazon EC2: Elastic Compute Cloud

**Amazon EC2** é o serviço de computação em nuvem da AWS que fornece capacidade de computação redimensionável na nuvem. É como ter acesso a um "supermercado de computadores" onde você pode escolher exatamente o tipo de máquina que precisa, usar pelo tempo necessário e pagar apenas pelo que consumir.

#### Analogia: EC2 como Locadora de Veículos
Imagine que você precisa de transporte, mas não quer comprar um carro:
- **Locadora tradicional** = Data center físico (você vai até lá, escolhas limitadas)
- **EC2** = Uber/app de transporte (sob demanda, variedade infinita, pagamento por uso)
- **Tipos de instância** = Diferentes categorias (econômico, executivo, van, caminhão)
- **AMI** = Configuração do veículo (combustível, GPS, ar-condicionado)
- **Key Pairs** = Chaves digitais para acessar
- **Security Groups** = Sistema de segurança e rastreamento

### Conceitos Fundamentais do EC2

#### O que é uma Instância?
Uma **instância EC2** é uma máquina virtual rodando na infraestrutura da AWS. Cada instância é um computador completo com:
- **CPU**: Processamento computacional
- **Memória RAM**: Para executar aplicações
- **Armazenamento**: Para dados e sistema operacional
- **Rede**: Para comunicação com internet e outros recursos

#### Ciclo de Vida de uma Instância
```
Pending → Running → Stopping → Stopped → Terminating → Terminated
    ↓         ↓         ↓         ↓           ↓
  Iniciando  Rodando  Parando  Parada    Destruída
```

- **Pending**: Instância sendo criada
- **Running**: Instância ativa e cobrando
- **Stopped**: Instância parada (não cobra compute, mas cobra storage)
- **Terminated**: Instância destruída permanentemente

### Tipos de Instâncias EC2

A AWS oferece diferentes "famílias" de instâncias otimizadas para casos de uso específicos:

#### 1. **General Purpose (Uso Geral)**
**Quando usar**: Aplicações web, pequenos bancos de dados, desenvolvimento

##### t3/t4g (Burstable Performance)
- **Conceito**: CPU com "créditos de burst"
- **Analogia**: Como um carro híbrido - economiza energia no dia a dia, mas tem potência extra quando precisa
- **Ideal para**: Aplicações com uso variável de CPU
- **Exemplo**: Site que tem picos de acesso em horários específicos

##### m5/m6i (Balanced)
- **Conceito**: Balanceamento entre CPU, memória e rede
- **Analogia**: Sedan familiar - bom para a maioria das situações
- **Ideal para**: Aplicações web de médio porte, microserviços
- **Exemplo**: API REST com tráfego constante

#### 2. **Compute Optimized (Otimizadas para Computação)**
**Quando usar**: Processamento intensivo, cálculos científicos, gaming

##### c5/c6i (High Performance CPU)
- **Conceito**: Máximo poder de processamento
- **Analogia**: Carro esportivo - toda potência focada na velocidade
- **Ideal para**: Simulações, análise de dados, servidores web de alta performance
- **Exemplo**: Sistema de processamento de imagens em tempo real

#### 3. **Memory Optimized (Otimizadas para Memória)**
**Quando usar**: Bancos de dados em memória, big data, cache

##### r5/r6i (High Memory)
- **Conceito**: Muita RAM disponível
- **Analogia**: Caminhão de carga - capacidade para carregar muito
- **Ideal para**: Bancos de dados, cache Redis, análise de big data
- **Exemplo**: Sistema de recomendações que mantém dados em memória

##### x1e (Extreme Memory)
- **Conceito**: Memória extrema (até 4TB de RAM)
- **Ideal para**: SAP HANA, Apache Spark, bancos de dados massivos

#### 4. **Storage Optimized (Otimizadas para Armazenamento)**
**Quando usar**: Data warehousing, sistemas de arquivos distribuídos

##### i3 (NVMe SSD)
- **Conceito**: Armazenamento super rápido
- **Ideal para**: Bancos NoSQL, data warehousing, cache distribuído

##### d3 (Dense HDD)
- **Conceito**: Muito espaço de armazenamento
- **Ideal para**: Processamento de logs, backup, arquivamento

#### 5. **Accelerated Computing (Computação Acelerada)**
**Quando usar**: Machine learning, renderização, mineração de criptomoedas

##### p4 (GPU para ML)
- **Conceito**: Processadores gráficos para cálculos paralelos
- **Ideal para**: Treinamento de IA, deep learning

##### g4 (GPU para Graphics)
- **Conceito**: Processamento gráfico profissional
- **Ideal para**: Renderização 3D, streaming de jogos

### AMI: Amazon Machine Images

#### O que é uma AMI?
**AMI** é como um "molde" ou "template" que contém tudo necessário para criar uma instância:
- Sistema operacional (Linux, Windows)
- Software pré-instalado
- Configurações iniciais
- Dados de aplicação (opcional)

#### Analogia: AMI como Receita de Bolo
- **Receita tradicional** = Instruções passo a passo
- **AMI** = Bolo pronto congelado que você só precisa "descongelar" (lançar)
- **Customização** = Você pode modificar a receita e criar sua própria AMI

#### Tipos de AMI:

##### 1. **Amazon Linux 2**
- **Características**: Otimizada para AWS, atualizações automáticas
- **Quando usar**: Aplicações novas, desenvolvimento
- **Vantagens**: Melhor performance na AWS, suporte oficial

##### 2. **Ubuntu**
- **Características**: Popular, grande comunidade
- **Quando usar**: Desenvolvimento web, aplicações Python/Node.js
- **Vantagens**: Muita documentação, pacotes atualizados

##### 3. **Windows Server**
- **Características**: Para aplicações Microsoft
- **Quando usar**: Aplicações .NET, Active Directory
- **Vantagens**: Integração com ecossistema Microsoft

##### 4. **Red Hat Enterprise Linux (RHEL)**
- **Características**: Focado em ambiente empresarial
- **Quando usar**: Aplicações críticas, compliance rigoroso
- **Vantagens**: Suporte comercial, certificações

#### Componentes de uma AMI:
- **Root Volume Template**: O "disco rígido" com SO e aplicações
- **Launch Permissions**: Quem pode usar esta AMI
- **Block Device Mapping**: Como os discos são organizados

### Security Groups: O Firewall Inteligente

#### O que são Security Groups?
**Security Groups** são como um "porteiro digital" que controla quem pode entrar e sair da sua instância EC2. É um firewall virtual que opera no nível da instância.

#### Analogia: Security Group como Portaria de Condomínio
- **Porteiro tradicional** = Firewall de hardware
- **Security Group** = Portaria inteligente com reconhecimento facial
- **Regras de entrada** = Lista de visitantes autorizados
- **Regras de saída** = Controle de quem pode sair e para onde

#### Características Importantes:

##### 1. **Stateful (Com Estado)**
- **O que significa**: Se você permite entrada, a saída correspondente é automática
- **Exemplo prático**: Se você permite HTTP (porta 80) de entrada, a resposta HTTP pode sair automaticamente
- **Vantagem**: Simplifica configuração, mais seguro

##### 2. **Deny by Default (Negação por Padrão)**
- **O que significa**: Tudo é bloqueado até você permitir explicitamente
- **Filosofia**: "Proibido até que seja permitido"
- **Vantagem**: Máxima segurança por padrão

##### 3. **Allow Rules Only (Apenas Regras de Permissão)**
- **O que significa**: Você só pode criar regras que permitem tráfego
- **Não existe**: Regra "negar acesso do IP X"
- **Como negar**: Simplesmente não criar regra de permissão

#### Configuração Típica de Security Group:

```
Nome: WebServer-SG
Descrição: Security Group para servidor web

Inbound Rules (Entrada):
┌─────────────┬──────┬─────────────┬─────────────────┐
│ Tipo        │ Porta│ Protocolo   │ Origem          │
├─────────────┼──────┼─────────────┼─────────────────┤
│ HTTP        │ 80   │ TCP         │ 0.0.0.0/0       │
│ HTTPS       │ 443  │ TCP         │ 0.0.0.0/0       │
│ SSH         │ 22   │ TCP         │ 203.0.113.0/24  │
└─────────────┴──────┴─────────────┴─────────────────┘

Outbound Rules (Saída):
┌─────────────┬──────┬─────────────┬─────────────────┐
│ Tipo        │ Porta│ Protocolo   │ Destino         │
├─────────────┼──────┼─────────────┼─────────────────┤
│ All Traffic │ All  │ All         │ 0.0.0.0/0       │
└─────────────┴──────┴─────────────┴─────────────────┘
```

**Explicação das regras:**
- **HTTP/HTTPS**: Permite acesso web de qualquer lugar (0.0.0.0/0)
- **SSH**: Permite acesso administrativo apenas da rede da empresa
- **Outbound All**: Permite que a instância acesse qualquer recurso externo

### Chaves SSH: Acesso Seguro sem Senhas

#### O que são SSH Key Pairs?
**SSH Key Pairs** são um par de chaves criptográficas que permitem acesso seguro às instâncias Linux sem usar senhas.

#### Analogia: SSH Keys como Chave e Fechadura Digital
- **Chave Privada** = Sua chave pessoal (nunca compartilhe)
- **Chave Pública** = A fechadura (pode ser copiada)
- **Processo** = Sua chave privada "abre" a fechadura pública

#### Como Funciona:
1. **Criação**: AWS gera um par de chaves
2. **Distribuição**: Chave pública vai para a instância, privada fica com você
3. **Conexão**: Seu cliente SSH usa a chave privada para provar identidade
4. **Autenticação**: Instância verifica usando a chave pública

#### Vantagens sobre Senhas:
- **Mais seguro**: Impossível de "adivinhar"
- **Não transmitido**: Chave privada nunca sai do seu computador
- **Revogável**: Pode remover acesso sem alterar senhas
- **Auditável**: Logs mostram qual chave foi usada

#### Processo de Conexão SSH:
```bash
# 1. Você executa
ssh -i minha-chave.pem ec2-user@ip-da-instancia

# 2. SSH client usa sua chave privada para criar "assinatura"
# 3. Instância verifica assinatura com chave pública
# 4. Se válida, conexão é estabelecida
```

### VPC: Virtual Private Cloud

#### O que é uma VPC?
**VPC** é sua rede privada virtual na AWS. É como ter seu próprio "bairro privado" na internet, onde você controla completamente o tráfego de rede.

#### Analogia: VPC como Condomínio Fechado
- **VPC** = O condomínio inteiro
- **Subnets** = Diferentes quadras/setores
- **Internet Gateway** = Portão principal de entrada
- **Route Tables** = Placas de sinalização interna
- **NAT Gateway** = Saída de serviço (funcionários podem sair, visitantes não entram)

#### Componentes Principais:

##### 1. **Subnets (Sub-redes)**
Divisões lógicas da sua VPC:

**Public Subnet:**
- **Características**: Tem acesso direto à internet
- **Uso típico**: Servidores web, load balancers
- **Analogia**: Lojas com frente para a rua

**Private Subnet:**
- **Características**: Sem acesso direto à internet
- **Uso típico**: Bancos de dados, servidores de aplicação
- **Analogia**: Depósitos nos fundos da loja

##### 2. **Internet Gateway (IGW)**
- **Função**: Permite comunicação entre VPC e internet
- **Analogia**: Portão principal do condomínio
- **Característica**: Altamente disponível e escalável

##### 3. **Route Tables (Tabelas de Roteamento)**
- **Função**: Definem para onde o tráfego de rede deve ir
- **Analogia**: GPS interno do condomínio
- **Exemplo**:
  ```
  Destino: 10.0.0.0/16 → Local (tráfego interno)
  Destino: 0.0.0.0/0  → Internet Gateway (tráfego externo)
  ```

##### 4. **NAT Gateway**
- **Função**: Permite que instâncias privadas acessem internet (apenas saída)
- **Analogia**: Saída de funcionários - podem sair, mas ninguém entra por ali
- **Uso**: Updates de software, APIs externas

#### Exemplo de Arquitetura VPC:
```
Internet
    │
    ▼
┌─────────────────────────────────────────┐
│ Internet Gateway                        │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│ VPC (10.0.0.0/16)                      │
│                                         │
│ ┌─────────────────┐ ┌─────────────────┐ │
│ │ Public Subnet   │ │ Private Subnet  │ │
│ │ (10.0.1.0/24)   │ │ (10.0.2.0/24)   │ │
│ │                 │ │                 │ │
│ │ [Web Server]    │ │ [Database]      │ │
│ │ [Load Balancer] │ │ [App Server]    │ │
│ └─────────────────┘ └─────────────────┘ │
└─────────────────────────────────────────┘
```

### Armazenamento EC2

#### EBS: Elastic Block Store
**EBS** é como um "HD externo" que você pode conectar à sua instância EC2.

##### Características:
- **Persistente**: Dados sobrevivem quando instância é parada
- **Flexível**: Pode aumentar tamanho e alterar tipo
- **Backup**: Snapshots automáticos ou manuais
- **Criptografia**: Dados podem ser criptografados

##### Tipos de EBS:

**gp3 (General Purpose SSD):**
- **Uso**: Aplicações gerais
- **Performance**: Boa relação custo-benefício
- **IOPS**: Configurável independentemente

**io2 (Provisioned IOPS SSD):**
- **Uso**: Aplicações críticas
- **Performance**: IOPS garantido
- **Custo**: Mais caro, mas previsível

**st1 (Throughput Optimized HDD):**
- **Uso**: Big data, data warehousing
- **Performance**: Alto throughput sequencial
- **Custo**: Mais barato para grandes volumes

#### Instance Store
**Instance Store** é como um "HD interno" que vem com a instância.

##### Características:
- **Temporário**: Dados perdidos quando instância para
- **Alta Performance**: Acesso direto ao hardware
- **Sem Custo Adicional**: Incluído no preço da instância
- **Não Persistente**: Use apenas para dados temporários

##### Quando Usar:
- Cache de aplicação
- Dados temporários de processamento
- Buffers de alta performance

### Modelos de Precificação EC2

#### 1. **On-Demand (Sob Demanda)**
- **Conceito**: Pague por hora/segundo de uso
- **Analogia**: Táxi - paga pela corrida
- **Vantagens**: Flexibilidade total, sem compromisso
- **Quando usar**: Desenvolvimento, testes, workloads imprevisíveis

#### 2. **Reserved Instances (Instâncias Reservadas)**
- **Conceito**: Compromisso de 1-3 anos com desconto
- **Analogia**: Plano de celular - paga mensalidade, usa quanto quiser
- **Desconto**: Até 75% comparado ao On-Demand
- **Quando usar**: Workloads estáveis e previsíveis

#### 3. **Spot Instances (Instâncias Spot)**
- **Conceito**: Aproveita capacidade ociosa da AWS
- **Analogia**: Passagem aérea de última hora - barata, mas pode ser cancelada
- **Desconto**: Até 90% do preço On-Demand
- **Risco**: AWS pode interromper com 2 minutos de aviso
- **Quando usar**: Processamento em lote, análise de dados

#### 4. **Dedicated Hosts (Hosts Dedicados)**
- **Conceito**: Servidor físico inteiro para você
- **Analogia**: Casa própria vs apartamento alugado
- **Uso**: Compliance, licenciamento específico
- **Custo**: Mais caro, mas controle total

### Segurança de Rede Avançada

#### Network ACLs (Access Control Lists)
- **Função**: Firewall no nível de subnet
- **Diferença do Security Group**: Stateless (precisa regra de entrada E saída)
- **Uso**: Camada adicional de segurança

#### VPC Flow Logs
- **Função**: Captura informações sobre tráfego IP
- **Uso**: Monitoramento, troubleshooting, segurança
- **Destino**: CloudWatch Logs, S3

#### AWS Systems Manager Session Manager
- **Função**: Acesso seguro sem SSH direto
- **Vantagem**: Não precisa abrir porta 22
- **Auditoria**: Todos os comandos são logados

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal diferença entre Security Groups e Network ACLs na AWS?

a) Security Groups são pagos, Network ACLs são gratuitos
b) Security Groups são stateful, Network ACLs são stateless
c) Security Groups funcionam apenas com EC2, Network ACLs com todos os serviços
d) Não há diferença, são sinônimos

**Gabarito: B**

**Justificativa:** Security Groups são stateful, o que significa que se você permitir tráfego de entrada, o tráfego de resposta é automaticamente permitido. Network ACLs são stateless, exigindo regras explícitas para entrada e saída. Esta é a diferença fundamental entre os dois mecanismos de segurança.

### Questão 2
Uma aplicação precisa de muito processamento de CPU para análise de dados. Qual família de instâncias EC2 seria mais adequada?

a) General Purpose (t3, m5)
b) Memory Optimized (r5, x1e)
c) Compute Optimized (c5, c6i)
d) Storage Optimized (i3, d3)

**Gabarito: C**

**Justificativa:** Instâncias Compute Optimized (c5, c6i) são especificamente projetadas para workloads que demandam alto desempenho de CPU, como análise de dados, computação científica e processamento intensivo. As outras famílias são otimizadas para diferentes recursos (memória, armazenamento, uso geral).

### Questão 3
O que acontece com os dados armazenados no Instance Store quando uma instância EC2 é parada (stopped)?

a) Os dados são automaticamente copiados para EBS
b) Os dados permanecem disponíveis quando a instância é reiniciada
c) Os dados são perdidos permanentemente
d) Os dados são arquivados no S3 automaticamente

**Gabarito: C**

**Justificativa:** Instance Store é armazenamento temporário fisicamente conectado ao servidor host. Quando uma instância é parada ou terminada, todos os dados no Instance Store são perdidos permanentemente. Para persistência de dados, deve-se usar volumes EBS.
